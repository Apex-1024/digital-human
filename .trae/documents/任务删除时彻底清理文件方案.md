# 任务删除时彻底清理文件方案

## Summary

当前 `DELETE /api/jobs/<job_id>` 接口删除任务时，只清理 `output/{job_id}/final/` 下的**文件**（不递归子目录），完全不清理 `uploads/` 下的 PPTX、PDF、MinerU 解析目录、PNG 切片目录，也不清理 `output/{job_id}/` 下的 `audio/`、`scenes/`、`talking_head_*/` 子目录。导致删除任务后磁盘残留大量文件。

本方案按照业界标准做法（事务性删除 + 路径推导 + 递归清理 + 容错）补全清理逻辑，确保删除任务后 `output/{job_id}/` 整个目录和 `uploads/` 下所有相关产物都被清理。

## Current State Analysis

### 现有删除逻辑缺陷（[app.py:145-158](file:///c:/Users/sun/Desktop/yuantu-test/digital-human/keling-batch/backend/app.py#L145-L158)）

```python
@app.route("/api/jobs/<job_id>", methods=["DELETE"])
def delete_job(job_id):
    job = store.get(job_id)
    if not job:
        return jsonify({"error": "任务不存在"}), 404
    if job.output_path:
        try:
            for f in Path(job.output_path).parent.iterdir():
                if f.is_file():
                    f.unlink()
        except Exception as e:
            log.warning(f"清理输出失败：{e}")
    store.delete(job_id)
    return jsonify({"ok": True})
```

**缺陷清单**：

| 应清理路径 | 推导方式 | 当前是否清理 |
|---|---|---|
| `output/{job_id}/` 整个目录 | 从 `job.output_path` 反推 `output/{job_id}/` | ❌ 只删 `final/` 下文件 |
| `output/{job_id}/audio/` | 同上 | ❌ |
| `output/{job_id}/scenes/` | 同上 | ❌ |
| `output/{job_id}/scenes/talking_head_{i}/` | 同上 | ❌ |
| `uploads/{stem}.pptx` | `job.pptx_path` | ❌ 完全不清理 uploads |
| `uploads/{stem}.pdf` | `job.pptx_path` 推 stem | ❌ |
| `uploads/_mineru_{stem}/` | `job.pptx_path` 推 stem | ❌ |
| `uploads/_slides_{stem}/` | `job.pptx_path` 推 stem | ❌ |

### Job 字段可用线索

- `job.pptx_path` — 原始 PPTX 路径（如 `uploads/d62a9c89_test.pptx`），可推 stem = `d62a9c89_test`
- `job.output_path` — 最终 MP4 路径（如 `output/e67e78c3/final/d62a9c89_test.mp4`），可推 `output/{job_id}/` = `Path(output_path).parent.parent`
- `job.scenes[i].image_path` — PNG 路径，可反推 `_slides_{stem}/`（但已有 stem 推导更直接）

### JobStore.delete（[core.py:720-723](file:///c:/Users/sun/Desktop/yuantu-test/digital-human/keling-batch/backend/core.py#L720-L723)）

只从内存缓存和 `jobs.json` 移除记录，不做任何文件清理。

## Proposed Changes

### 设计原则（业界标准做法）

1. **集中清理函数**：在 `core.py` 新增 `_cleanup_job_files(job: Job)` 工具函数，由 `app.py` 在 `delete_job` 中调用。`JobStore` 保持纯数据存储职责，不混入文件操作。
2. **路径推导**：从 `job.pptx_path` 和 `job.output_path` 两条线索推导所有产物路径，避免维护额外字段。
3. **递归删除**：目录用 `shutil.rmtree(path, ignore_errors=True)`，文件用 `Path.unlink(missing_ok=True)`。
4. **容错**：单个路径清理失败不影响其他路径，记录 warning 日志但不抛异常。
5. **顺序**：先清理文件，再 `store.delete(job_id)`。若文件清理失败仍删除记录（避免残留无主任务），但记录 warning。
6. **运行中任务保护**：删除前检查 `job.status`，若为 `running` 返回 409 Conflict，避免删除运行中任务的文件导致崩溃。

### 改动 1：core.py 新增 `_cleanup_job_files` 函数

**文件**：`c:\Users\sun\Desktop\yuantu-test\digital-human\keling-batch\backend\core.py`

**位置**：在 `JobStore` 类定义之前（约行 630 附近，`Scene` 类之后），作为模块级工具函数。

**新增代码**：

```python
def _cleanup_job_files(job: "Job") -> List[str]:
    """清理任务相关的所有文件和目录。

    从 job.pptx_path 和 job.output_path 推导所有产物路径并删除。
    单个路径失败不影响其他路径，返回已清理路径列表供日志记录。

    清理范围：
    - uploads/{stem}.pptx              原始上传文件
    - uploads/{stem}.pdf               PPT 转 PDF 产物
    - uploads/_mineru_{stem}/          MinerU 解析结果目录
    - uploads/_slides_{stem}/          PDF 转 PNG 切片目录
    - output/{job_id}/                 输出工作目录（含 audio/ scenes/ final/ talking_head_*/）
    """
    import shutil
    cleaned: List[str] = []

    # 1) 从 pptx_path 推导 uploads/ 下所有产物
    if job.pptx_path:
        pptx = Path(job.pptx_path)
        stem = pptx.stem  # 如 d62a9c89_test
        uploads_dir = pptx.parent

        upload_artifacts = [
            pptx,                                  # 原始 PPTX
            uploads_dir / f"{stem}.pdf",           # PPT 转 PDF
            uploads_dir / f"_mineru_{stem}",       # MinerU 解析目录
            uploads_dir / f"_slides_{stem}",       # PNG 切片目录
        ]
        for p in upload_artifacts:
            try:
                if p.is_dir():
                    shutil.rmtree(p, ignore_errors=False)
                    cleaned.append(str(p))
                elif p.exists():
                    p.unlink(missing_ok=False)
                    cleaned.append(str(p))
            except Exception as e:
                log.warning(f"清理失败 {p}: {e}")

    # 2) 从 output_path 推导 output/{job_id}/ 整个目录
    if job.output_path:
        out_file = Path(job.output_path)
        # output_path = output/{job_id}/final/{stem}.mp4
        # 工作目录 = output/{job_id}/
        job_work_dir = out_file.parent.parent
        if job_work_dir.exists() and job_work_dir.is_dir():
            # 安全检查：必须是 OUTPUT_DIR 的直接子目录，避免误删
            try:
                if job_work_dir.parent == OUTPUT_DIR:
                    shutil.rmtree(job_work_dir, ignore_errors=False)
                    cleaned.append(str(job_work_dir))
                else:
                    log.warning(
                        f"拒绝清理非 OUTPUT_DIR 子目录: {job_work_dir}"
                    )
            except Exception as e:
                log.warning(f"清理失败 {job_work_dir}: {e}")

    return cleaned
```

**关键设计点**：
- **安全检查**：`job_work_dir.parent == OUTPUT_DIR` 防止 `output_path` 被篡改后误删其他目录
- **shutil.rmtree(ignore_errors=False)** + try/except：记录具体失败原因，而非静默吞掉
- **返回 cleaned 列表**：供调用方记录日志，便于调试

@app.route("/api/jobs/<job_id>", methods=["DELETE"])
def delete_job(job_id):
    job = store.get(job_id)
    if not job:
        return jsonify({"error": "任务不存在"}), 404

    # 运行中任务拒绝删除，避免文件被占用导致崩溃
    if job.status in ("running", "pending"):

    if cleaned:
        log.info(f"任务 {job_id} 已清理 {len(cleaned)} 个文件/目录: {cleaned}")
    store.delete(job_id)
    return jsonify({"ok": True, "cleaned": len(cleaned)})

- 新增运行中任务保护（409 Conflict）

## Assumptions & Decisions

### 假设
1. **stem 推导可靠**：`job.pptx_path` 始终指向 `uploads/{stem}.pptx`，PDF/MinerU/slides 目录都遵循 `{stem}` 命名规则（已通过代码探索验证）
2. **output 目录结构稳定**：`output/{job_id}/{audio,scenes,final}/` 结构不会变（已通过 [core.py:1340-1345](file:///c:/Users/sun/Desktop/yuantu-test/digital-human/keling-batch/backend/core.py#L1340-L1345) 验证）
3. **uploads 扁平结构**：uploads/ 下没有 job_id 子目录，所有产物用 `{stem}` 前缀区分（已通过代码和磁盘验证）

### 决策
1. **不新增 Job 字段**：不为 `pdf_path`、`mineru_dir`、`slides_dir` 新增 Job 字段，避免 dataclass 变更和 jobs.json 向后兼容问题。用 stem 推导更轻量。
2. **清理函数放 core.py 而非 app.py**：core.py 已有所有路径常量（OUTPUT_DIR、UPLOAD_DIR）和 Job 定义，放这里内聚性更好。app.py 只负责 HTTP 路由。
3. **保留运行中任务保护**：删除运行中任务会导致 ffmpeg/SadTalker 进程写文件失败崩溃，必须保护。pending 状态也保护（可能即将启动）。
4. **彻底删除原始 PPTX**：用户明确要求"output和uploads两个文档生成的残留文件都没有被删除"，即 uploads/ 也要清理，包括原始 PPTX。
5. **不清理全局共享资源**：assets/、AVATAR_DIR、BGM_DIR、_mineru_patch/、jobs.json 不在清理范围。

## Verification Steps

### 1. 手动验证清理效果

提交一个新任务并完成后：

```powershell
# 记录删除前的文件

Get-ChildItem output\{job_id}\ -Recurse | Measure-Object

# 调用删除接口
curl -X DELETE http://127.0.0.1:5000/api/jobs/{job_id}

# 验证清理后
Get-ChildItem uploads\ -Recurse | Measure-Object  # 应减少 4 项（pptx/pdf/_mineru_/_slides_）
Test-Path output\{job_id}  # 应为 False
```

### 2. 边界场景验证

| 场景 | 预期行为 |
|---|---|
| 删除运行中任务 | 返回 409，不删任何文件 |
| 删除已完成任务 | 清理 uploads/ 4 项 + output/{job_id}/ 整个目录 |
| 删除失败任务（无 output_path） | 仅清理 uploads/ 产物 |
| 删除不存在的任务 | 返回 404 |
| 某个文件被占用 | 该文件 warning 日志，其他文件正常清理，记录仍删除 |
| jobs.json 记录 | 任务记录被移除 |

### 3. 日志验证

删除成功后服务端日志应出现：

```
任务 {job_id} 已清理 N 个文件/目录: ['uploads/xxx.pptx', 'uploads/xxx.pdf', ...]
```

### 4. 回归验证

- 删除任务后再次上传同文件名，不应冲突
- jobs.json 中该任务记录消失
- 前端任务列表刷新后该任务消失
